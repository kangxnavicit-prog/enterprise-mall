# 企业内部商城 PRD（Product Requirements Document）

**项目名称**：enterprise-mall  
**版本**：v1.0  
**作者**：许清楚（Xu），产品经理

## 1. 产品目标

### 1.1 核心价值主张

企业内部商城为企业员工提供统一的福利兑换与内部商品采购平台，降低人工发放成本，提升员工福利体验与满意度；为管理员提供商品、订单、用户的全生命周期管理能力，使企业福利运营数字化、可追溯。

### 1.2 目标用户

| 用户角色 | 描述 | 核心诉求 |
|---------|------|---------|
| **员工（普通用户）** | 企业全体在职员工 | 便捷浏览商品、用积分兑换福利、查看订单状态 |
| **管理员** | HR、行政或运营人员 | 高效管理商品与库存、处理订单发货、查看运营数据 |

### 1.3 成功指标（OKR/KPI）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 员工活跃率 | ≥ 60% MAU/注册用户 | 上线后第 3 个月起 |
| 订单转化率 | ≥ 40% 浏览→下单 | 商品详情页到提交订单 |
| 积分消耗率 | ≥ 70% 月度积分发放量被兑换 | 衡量福利吸引力 |
| 管理员操作效率 | 订单处理时效 ≤ 1 个工作日 | 发货操作完成率 |
| 系统可用性 | ≥ 99.5% | 服务 SLA |

## 2. 用户故事

### 2.1 员工端

| # | 用户故事 |
|---|---------|
| US-01 | As an **员工**，I want to **使用企业账号登录商城**，so that **可以安全地访问个人积分和下单权限** |
| US-02 | As an **员工**，I want to **按分类/关键词搜索商品**，so that **能快速找到我需要的福利商品** |
| US-03 | As an **员工**，I want to **将商品加入购物车并调整数量**，so that **可以一次性兑换多件商品** |
| US-04 | As an **员工**，I want to **提交订单并看到订单确认信息**，so that **确认兑换成功、了解后续流程** |
| US-05 | As an **员工**，I want to **在「我的订单」中查看订单状态**，so that **随时了解商品是否已发货/到达** |
| US-06 | As an **员工**，I want to **在个人中心查看积分余额与消费记录**，so that **合理规划积分使用** |

### 2.2 管理员端

| # | 用户故事 |
|---|---------|
| US-07 | As an **管理员**，I want to **增删改查商品信息与库存**，so that **保持商城商品信息准确、实时** |
| US-08 | As an **管理员**，I want to **查看所有订单并执行发货/完成操作**，so that **高效处理员工兑换请求** |
| US-09 | As an **管理员**，I want to **管理用户积分（发放/扣减/查看记录）**，so that **灵活运营福利活动** |
| US-10 | As an **管理员**，I want to **在数据看板上查看订单量、销售额、热销商品**，so that **做出采购和运营决策** |

## 3. 需求池（按优先级）

### P0 — 核心功能（Must Have，MVP 范围）

| ID | 功能模块 | 需求描述 | 验收标准 |
|----|---------|---------|---------|
| P0-01 | 员工登录/登出 | JWT 认证，支持账号+密码登录；Token 刷新；登出清除本地 Token | 登录成功后跳转首页；Token 过期自动重定向登录页 |
| P0-02 | 商品列表页 | 支持分类筛选、关键词搜索、分页（20 条/页） | 筛选结果实时更新，分页无刷新 |
| P0-03 | 商品详情页 | 展示商品图片（多图轮播）、名称、描述、积分价格、库存状态 | 库存为 0 时禁止加购 |
| P0-04 | 购物车 | 增/删商品，修改数量，实时计算总积分；库存不足时提示 | 购物车数据持久化（页面刷新不丢失） |
| P0-05 | 提交订单 | 选择收货地址，确认商品与积分，提交后扣减积分与库存 | 积分不足时提示并阻止提交；库存并发扣减安全 |
| P0-06 | 我的订单列表 | 按状态分 Tab：待发货 / 已发货 / 已完成；显示订单号、商品、时间 | 状态更新后列表实时反映 |
| P0-07 | 管理员：商品 CRUD | 新增/编辑/下架商品；上传多图；设置分类、积分价格、库存 | 下架商品不在前台展示；编辑保存实时生效 |
| P0-08 | 管理员：订单状态管理 | 批量或单条标记「发货」（填写快递信息）或「完成」 | 状态流转：待发货→已发货→已完成，不可逆 |

### P1 — 重要功能（Should Have，上线后第二期）

| ID | 功能模块 | 需求描述 |
|----|---------|---------|
| P1-01 | 商品分类管理 | 管理员后台支持分类的增删改排序；前台分类导航联动 |
| P1-02 | 用户积分/余额体系 | 管理员批量发放积分；员工查看积分明细；下单自动扣减 |
| P1-03 | 数据看板 | 订单总量/销售额趋势图、热销商品 Top10、积分发放与消耗统计 |
| P1-04 | 商品评论/评分 | 已完成订单可评价；展示商品平均分和评价列表 |

### P2 — 可选增强（Nice to Have，视资源排期）

| ID | 功能模块 | 需求描述 |
|----|---------|---------|
| P2-01 | 消息通知 | 订单状态变更（站内消息 + 可选邮件）；系统公告推送 |
| P2-02 | 收藏夹 | 员工收藏商品，快速进入购买流程 |
| P2-03 | 优惠券/活动 | 限时折扣、积分翻倍活动；优惠券码兑换 |

## 5. 数据模型草图

### 5.1 User（用户）

```
User {
  id            UUID          PK
  employee_id   VARCHAR(50)   UNIQUE  -- 工号
  username      VARCHAR(100)  UNIQUE
  password_hash VARCHAR(255)
  role          ENUM('employee', 'admin')
  department    VARCHAR(100)
  points        INTEGER       DEFAULT 0  -- 当前积分余额
  avatar_url    VARCHAR(500)
  is_active     BOOLEAN       DEFAULT true
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
}
```

### 5.2 Category（商品分类）

```
Category {
  id          UUID
  name        VARCHAR(100)   -- 分类名称
  icon        VARCHAR(500)   -- 分类图标 URL
  sort_order  INTEGER        -- 排序权重
  is_active   BOOLEAN        DEFAULT true
  created_at  TIMESTAMP
}
```

### 5.3 Product（商品）

```
Product {
  id            UUID
  category_id   UUID          FK -> Category.id
  name          VARCHAR(200)
  description   TEXT
  images        JSONB         -- ["url1", "url2", ...]
  points_price  INTEGER       -- 积分价格
  stock         INTEGER
  status        ENUM('on_sale', 'off_sale')
  avg_rating    DECIMAL(2,1)  DEFAULT 0  -- 冗余字段，P1
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
}
```

### 5.4 CartItem（购物车条目）

```
CartItem {
  id          UUID
  user_id     UUID          FK -> User.id
  product_id  UUID          FK -> Product.id
  quantity    INTEGER
  created_at  TIMESTAMP
  updated_at  TIMESTAMP
  -- 唯一约束：(user_id, product_id)
}
```

### 5.5 Order / OrderItem（订单）

```
Order {
  id              UUID
  order_no        VARCHAR(32)   UNIQUE  -- ORD-YYYYMMDD-XXXXX
  user_id         UUID          FK -> User.id
  total_points    INTEGER
  status          ENUM('pending', 'shipped', 'completed', 'cancelled')
  address         JSONB         -- {name, phone, address}
  tracking_no     VARCHAR(100)  -- 快递单号（发货后填写）
  remark          TEXT
  created_at      TIMESTAMP
  shipped_at      TIMESTAMP
  completed_at    TIMESTAMP
}

OrderItem {
  id             UUID
  order_id       UUID          FK -> Order.id
  product_id     UUID          FK -> Product.id
  product_name   VARCHAR(200)  -- 快照
  product_image  VARCHAR(500)  -- 快照
  points_price   INTEGER       -- 下单时快照
  quantity       INTEGER
}
```

### 5.6 PointRecord（积分记录）

```
PointRecord {
  id          UUID
  user_id     UUID          FK -> User.id
  type        ENUM('grant', 'deduct', 'refund')  -- 发放/消费/退还
  points      INTEGER       -- 正为增加，负为扣减
  balance     INTEGER       -- 操作后余额（快照）
  reason      VARCHAR(200)  -- 来源说明（如：订单 ORD-xxx 消费）
  order_id    UUID          FK -> Order.id  NULLABLE
  operator_id UUID          FK -> User.id  NULLABLE  -- 管理员操作时记录
  created_at  TIMESTAMP
}
```

### 5.7 Review（评论，P1）

```
Review {
  id          UUID
  user_id     UUID          FK -> User.id
  product_id  UUID          FK -> Product.id
  order_id    UUID          FK -> Order.id
  rating      INTEGER       -- 1~5
  content     TEXT
  images      JSONB         -- 评价图片
  created_at  TIMESTAMP
  -- 唯一约束：(user_id, order_id, product_id)
}
```

## 6. 待确认问题（Open Questions）

| # | 问题 | 选项 / 影响 | 优先级 |
|---|------|------------|--------|
| OQ-01 | **积分发放机制** | 按月自动发放（金额/规则？）还是管理员手工批量导入？ | P0 相关 |
| OQ-02 | **收货地址** | 是否需要员工自填地址？还是公司统一派发到工位/前台？若统一派发可简化流程 | P0 相关 |
| OQ-03 | **账号来源** | 员工账号是否通过 LDAP/SSO（如企业微信、钉钉）同步？还是商城自建账号体系？ | P0 相关 |
| OQ-04 | **库存并发策略** | 热门商品秒杀场景是否需要悲观锁 / Redis 库存预扣？还是简单乐观锁即可？ | P0 相关 |
| OQ-05 | **订单取消** | 员工是否可以自主取消「待发货」订单？取消后积分如何退还？ | P1 相关 |
| OQ-06 | **管理员权限分级** | 是否需要超级管理员 vs. 普通管理员（如只能操作自己部门的订单）？ | P1 相关 |
| OQ-07 | **数据隔离** | 是否为多租户（多个企业共用一套系统），还是单租户部署？ | 架构影响大 |
